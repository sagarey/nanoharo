## Critical Gap: RUNNER-04 Incomplete

### Problem

**RUNNER-04** 要求 IPC follow-up 消息机制改为进程内直接传递。

实现进度：

- ✅ **Write side:** `GroupQueue.sendMessage()` (`group-queue.ts:154`) 将 follow-up 文本推入 `state.pendingFollowUps` 内存数组
- ❌ **Read side:** 没有任何代码消费 `pendingFollowUps`。`runForGroup`、`processGroupMessages`、`runAgent`、`runInProcessAgent` 均无 drain 循环

### Impact

用户在 agent 处理消息期间发送的追加消息：
1. 被 `sendMessage()` 接受（返回 `true`）
2. 推入 `state.pendingFollowUps`
3. 永远不会通过 `session.send()` 送达 Claude SDK
4. 在该 group 下一次 `runForGroup` 调用前一直积累（但依然不会被消费）

### Root Cause

Phase 01 verifier 通过静态分析确认了 write side（`pendingFollowUps.push()`），但未检测 read side（`pendingFollowUps` 消费者）是否存在。Integration checker 通过跨文件分析发现了这一缺口。

### Fix Required

在 `runForGroup` 的 `session.stream()` 完成后，需要添加 drain 循环：
```typescript
// After stream() ends, drain any pending follow-ups
while (state.pendingFollowUps.length > 0 && state.session) {
  const followUp = state.pendingFollowUps.shift()!;
  await runInProcessAgent(group, { ...input, prompt: followUp, sessionId: capturedSessionId }, ...);
}
```
或在 `runInProcessAgent` 中直接用 `session.send()` API 发送追加消息（取决于 V2 SDK 的 multi-turn API）。
